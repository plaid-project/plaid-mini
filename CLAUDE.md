# PLAID Reduced Stack

## Architecture

```
Host (Ubuntu, /dev/kvm)
└── Firecracker microVM
    └── Alpine Linux guest (Red Green) — 172.16.0.2
        └── crun container runtime
            └── Bill (Yocto Zeus x86-64 OCI image)
                └── Python 3.7 + bash
```

## From-Scratch Build

```bash
make build-all    # builds everything: builder, bill, assets, kernel, rootfs
make up           # boots Firecracker VM
make load-bill    # loads Bill OCI image into guest, starts container
make shell        # drops into bash inside Bill
```

`make build-all` runs these in order:

1. **`builder`** — builds `vpanel-yocto-zeus-builder:latest` Docker image
   from `bill/Dockerfile.builder` (Ubuntu 18.04 with Yocto deps)
2. **`bill`** — runs Yocto build inside builder container, imports result as
   `vpanel-bill:latest`. First cold build takes hours; subsequent builds use
   sstate cache (Docker volumes `vpanel-base-x86_yocto-downloads` and
   `vpanel-base-x86_yocto-sstate`, created automatically on first run).
3. **`assets`** — downloads Firecracker binary
4. **`kernel`** — downloads prebuilt vmlinux (will become a local build)
5. **`rootfs`** — builds Alpine ext4 rootfs, generates SSH keys

## Key Technical Details

**Firecracker is not Docker.** It's a VMM configured via REST API over a Unix
socket. `launch.sh` starts the process and PUTs boot-source, drives,
network-interfaces, machine-config, then InstanceStart.

**Host networking:** tap0 at 172.16.0.1/24, guest eth0 at 172.16.0.2/24.
NAT via iptables. Setup/teardown require sudo.

**Yocto Zeus only builds on Ubuntu 18.04.** The builder container handles this.

**`IMAGE_INSTALL_append` needs a leading space:** `" python3"` not `"python3"`.

**Yocto sstate volumes:** `docker compose run` creates these as empty Docker
volumes on first run. First build is cold (hours). After that, sstate makes
rebuilds fast.

**The guest rootfs is ext4.** Built via: Docker image → `docker export` →
extract into mounted ext4 file. Needs sudo for mount/mknod.

**SSH keypair:** ed25519, generated by `build-rootfs/build.sh`. Public key
baked into rootfs, private key at `keys/rg_key`.

## File Layout

```
Makefile                  # all build and runtime targets
get-firecracker.sh        # downloads firecracker binary + prebuilt kernel
setup-host.sh             # tap0 + iptables (sudo)
teardown-host.sh          # reverse of setup-host
launch.sh                 # starts firecracker via REST API
stop.sh                   # kills firecracker process
wait-for-ssh.sh           # polls until guest SSH is up
assets/                   # generated: firecracker, vmlinux, alpine-rootfs.ext4
keys/                     # generated: rg_key, rg_key.pub
kernel/                   # future: custom kernel config and build
build-rootfs/
  Dockerfile.rootfs       # Alpine guest image definition
  build.sh                # builds ext4 rootfs + SSH keys
bill/
  Dockerfile.builder      # Yocto builder image (Ubuntu 18.04)
  docker-compose.yml      # runs builder with sstate volumes
  build.sh                # kicks off Yocto build
  build-inner.sh          # runs inside builder container
  import.sh               # imports Yocto output as Docker image
  conf/
    local.conf            # Yocto config (machine, packages)
    bblayers.conf          # Yocto layer config
  build-output/           # generated: rootfs tarball
```

## Future Work

**Custom Firecracker kernel:** The `kernel/` directory is reserved for a
custom kernel build with additional drivers. When ready, add a defconfig to
`kernel/`, a build script, and change the `assets/vmlinux` Makefile rule to
build from source instead of downloading.

**Cloud packaging:** Target is Packer → AMI (or GCE image) on a `.metal` or
nested-virt-capable instance type. Firecracker requires `/dev/kvm`. The image
would have all assets pre-baked with `make up && make load-bill` run at boot
via cloud-init or systemd.
